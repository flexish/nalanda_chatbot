Section 01

Executive Summary

The Nalanda Multimodal Agentic RAG System is an enterprise-grade, AI-powered document intelligence platform built specifically for the Nalanda Mahavihara archaeological knowledge base. It enables users to query large collections of PDF documents — including scanned books, excavation reports, conservation records, and research papers — using natural language and receive accurate, cited, and verified answers in real time.

Unlike a standard chatbot, this system uses an Agentic RAG (Retrieval-Augmented Generation) pipeline that actively retrieves evidence from your documents before generating any answer, verifies the answer against that evidence, and can automatically retry with a refined search if the first attempt is insufficient. It also understands and retrieves images, photographs, maps, and figures embedded in PDFs — making it truly multimodal.

🎯

Purpose

Give researchers, historians, and administrators instant, accurate answers from thousands of pages of Nalanda-related documents without manual searching.

🔒

Access Control

Role-based access: regular users can chat, admins can upload documents, manage the knowledge base, and manage user accounts.

⚡

Real-Time

Answers stream word-by-word as the AI generates them, providing immediate feedback without waiting for the full response.

🧠

Self-Verifying

Every answer is automatically checked for factual grounding. Unverified answers are flagged with a visible warning badge.

Key Differentiators

  • Answers come only from your uploaded documents — no hallucinated facts from general AI training data
  • Understands follow-up questions using conversation history (e.g., "Tell me more about it")
  • Retrieves images and photographs from PDFs and displays them alongside text answers
  • Uses 7 advanced RAG techniques stacked together for maximum retrieval accuracy
  • Admins can register web URL sources — page content is fetched fresh at query time and given directly to the AI alongside PDF context, exactly like ChatGPT or Claude browsing
  • Fully configurable via environment variables — switch LLM providers, tune chunk sizes, enable web search, all without code changes
  • Built on open standards — all APIs are REST/SSE, no vendor lock-in for the interface layer
Section 02

System Architecture

The system is composed of four layers that work together from document ingestion through to the user's chat interface.

Architecture Layers

📄

Layer 1 — Ingestion Layer

PDF files are parsed using Unstructured + Tesseract OCR. Text, tables, and images are extracted, cleaned, and summarized by an LLM. Summaries are embedded and stored in ChromaDB; full content is stored in a docstore.

🔍

Layer 2 — Retrieval Layer

User questions go through query rewriting, multi-query expansion, and HyDE before retrieval. Results from keyword + semantic search are merged, deduplicated, and re-ranked by a cross-encoder model.

🤖

Layer 3 — Generation & Verification Layer

Retrieved context is sent to the LLM (GPT-4o or Claude) to generate an answer. A second LLM call verifies the answer is grounded in the retrieved text. If not, an iterative retrieval hop is triggered.

🌐

Layer 4 — Presentation Layer

FastAPI serves a REST + SSE API. The frontend (vanilla HTML/JS/CSS) renders streamed tokens, images, verification badges, and manages user sessions and admin controls.

Deployment

The system runs as a single Python process (FastAPI with Uvicorn). The frontend is served as static files from the same server. No separate frontend build step or Node.js is required. ChromaDB and the docstore are file-based and stored locally on the server.

Single-server deployment: The entire system — API, frontend, vector database, and docstore — runs on one machine. For production scale, ChromaDB can be replaced with a hosted vector database and the FastAPI app can be containerized with Docker.

4.1 PDF Parsing & OCR

Unstructured (with strategy="hi_res") processes each PDF page using computer vision to accurately detect text columns, table boundaries, and embedded images — even in complex multi-column layouts common in academic papers and archaeological reports.

Tesseract OCR is applied to scanned pages where text is rendered as an image rather than machine-readable characters. This ensures documents like old survey reports or photographed manuscripts are also searchable.

The chunking strategy is "by_title" — content is split at section headings rather than at arbitrary character limits, preserving semantic meaning within each chunk. Configurable parameters:

  • CHUNK_MAX_CHARS — maximum characters per chunk (default: 10,000)
  • CHUNK_COMBINE_UNDER — merge adjacent chunks smaller than this (default: 2,000)
  • CHUNK_NEW_AFTER — start a new chunk after this many characters (default: 6,000)

4.2 Text Cleaning

Raw PDF text contains artifacts that hurt search quality. The system automatically cleans:

  • Hyphenated line-breaks — "discov-\nery" is joined back to "discovery" so the complete word is searchable
  • Excessive blank lines — 3+ consecutive newlines collapsed to one paragraph break
  • Repeated spaces — multiple spaces/tabs normalized to single space
  • Form-feed characters — page-break control characters (0x0C) removed

4.3 Summarization & Embedding

Rather than embedding raw text chunks directly (which would make long chunks difficult to match against short user queries), the system uses the Multi-Vector Retriever pattern:

Summary stored in ChromaDB (searchable)

The LLM generates a concise summary of each chunk. This summary is embedded using OpenAI's text-embedding-3-small and stored as a vector in ChromaDB. User queries are matched against these summaries.

Full content stored in docstore (returned)

The original full-length text, table HTML, or base64 image is stored separately in docstore.pkl. When a summary matches a query, the full parent document is fetched and sent to the LLM as context.

Image summaries are generated using GPT-4o's vision capability — the image is base64-encoded and sent to the LLM which describes what it shows in the context of Nalanda Mahavihara.

4.4 Multi-Vector Storage

Every stored entry carries rich metadata in both ChromaDB and the docstore:

  • source — path to the originating PDF file
  • kind — document type: text, table, or image
  • page_number — PDF page where this chunk originated (used for proximity-based image retrieval)
  • doc_id — UUID linking the summary vector to its parent document
Re-indexing: If the same PDF is indexed again (e.g., after an update), the system automatically removes all existing entries for that file before adding new ones — preventing duplicate chunks in the knowledge base.
Section 05

Retrieval Augmented Generation (RAG) Pipeline

The RAG pipeline is the core intelligence of the system. It transforms a user's question into a high-quality, evidence-backed answer through 7 stacked advanced techniques, orchestrated as a LangGraph state machine.

5.1 Query Rewriting

Before any retrieval, the user's question is sent to the LLM for rewriting. This serves two purposes:

  • Pronoun resolution — "Tell me more about it" becomes "Tell me more about Nalanda's excavation history" using the conversation history
  • Search optimization — colloquial questions are rephrased into terminology more likely to appear in academic documents (e.g., "Who built it?" → "Nalanda Mahavihara founder construction history")

Technology: OpenAI GPT-4o / Anthropic Claude via LangChain's ChatOpenAI / ChatAnthropic

5.2 Multi-Query Retrieval

Instead of running one search, the system generates 3 query variants and runs retrieval for each:

Query 1 — Rewritten

The optimized rewrite of the original question (from §5.1).

Query 2 — Alternative

A different phrasing likely to retrieve different relevant document chunks.

Query 3 — HyDE

A hypothetical answer snippet (see §5.3) used as a search query.

Results from all three queries are merged with deduplication — if the same document chunk appears in multiple results, it is included only once.

Technology: LangChain + ChromaDB similarity_search_with_relevance_scores + in-memory keyword scanning

5.3 HyDE — Hypothetical Document Embeddings

HyDE is a technique that improves semantic retrieval by searching with an answer instead of a question. The LLM generates a hypothetical 1-2 sentence passage that would appear in a real document answering the question. This hypothetical passage is then embedded and used for vector search.

Why this helps: In embedding space, a hypothetical answer is closer to real document passages than the question itself — because both are written in the same "answer" style. This retrieves relevant chunks that a plain question embedding would miss.

Technology: OpenAI GPT-4o generates the hypothesis; text-embedding-3-small embeds it; ChromaDB finds nearest neighbors

5.4 Re-Ranking (Cross-Encoder)

After multi-query retrieval, the merged candidate documents are re-ranked using a cross-encoder model. Unlike the embedding-based retrieval (which encodes query and document separately), the cross-encoder reads the question and document together to produce a much more accurate relevance score.

Model used: cross-encoder/ms-marco-MiniLM-L-6-v2 — a lightweight, fast cross-encoder trained on MS MARCO passage ranking. Runs locally on CPU; no API cost.

The top-K documents after re-ranking are kept and passed to the LLM for answer generation.

5.5 CRAG — Corrective RAG / Web Search Fallback

After re-ranking, the system scores the quality of retrieved documents. If the total retrieved text volume is below a threshold (suggesting the question is not covered by the indexed documents), it automatically triggers a DuckDuckGo web search and adds the top results as additional context.

This is controlled by the ENABLE_WEB_SEARCH environment variable (off by default). When enabled, the system seamlessly augments local document knowledge with live web information.

Technology: duckduckgo-search Python library

5.6 Self-RAG Verification

After the LLM generates an answer, a second LLM call acts as a quality checker. It reads the retrieved context, the question, and the generated answer, then answers two questions:

  • GROUNDED: yes/no — Is the answer supported by the retrieved document context?
  • KEEP_IMAGE: yes/no — Is the retrieved image actually relevant to the question asked?
  • REASON — One sentence explaining the judgment

If GROUNDED: no, the answer is flagged with a "may not be supported" note visible to the user. If KEEP_IMAGE: no, the image is removed from the response.

5.7 Iterative Retrieval

If an answer is flagged as unverified and the system has not yet reached the maximum hop limit (MAX_ITERATIVE_HOPS, default: 1), the LangGraph pipeline routes to an iterate node instead of ending:

Section 06

Multimodal Capabilities

The system handles not just text but also tables and images embedded in PDFs, making it fully multimodal.

📝

Text Chunks

Paragraphs and sections extracted from PDFs. Summarized by LLM, embedded, and stored. Retrieved semantically and by keyword overlap.

📊

Tables

Tables extracted with HTML structure preserved. Summarized by LLM so their content is semantically searchable even when the question doesn't match table headers exactly.

🖼️

Images

Photographs, maps, diagrams, and figures extracted as base64. GPT-4o Vision describes them in Nalanda context. Retrieved by semantic caption matching or page proximity to relevant text.

Image Retrieval Strategy (4-tier)

When a user asks an image-related question ("show me", "display", "photo of"), the system attempts image retrieval in order of confidence:

Tier 1 — Keyword Image Scan

Scans all image captions and source filenames for exact token matches with the question. Highest precision — returns images whose captions directly mention the query entities.

Tier 2 — Page Proximity

Finds images on the same page (±2 pages) as the best-matching text chunks. Uses the text's page_number metadata stored during ingestion. Very reliable for figures referenced in text.

Tier 3 — Semantic Caption Search

Runs semantic similarity search over image captions using the query embedding. Finds conceptually related images even when exact words don't match.

Tier 4 — Weak Semantic Fallback

Lowers the relevance threshold for a final attempt. Used only when the previous three tiers return nothing. Prevents completely empty image responses.

Image Validity Check

Every image is validated as a renderable image (using Pillow's Image.verify()) before it is stored or returned. Corrupted or partial extractions are silently discarded, ensuring only clean images reach the user's browser.

Requested Image Limit

Users can request a specific number of images in their question ("show me 3 images of the stupa", "show all images of the excavation"). The system parses the number from the question and applies it as a limit on the images returned. Defaults to a maximum of 6 images per response to avoid overloading the interface.

Section 07

Real-Time Streaming

Rather than waiting for the full answer to be generated (which can take 5-15 seconds for long responses), the system streams the answer to the browser token by token as the LLM generates it — similar to how ChatGPT displays its responses.

How Streaming Works

1. SSE Connection

The browser opens a Server-Sent Events connection to POST /api/chat/stream. This is a long-lived HTTP connection that the server pushes events through.

2. Token Events

Each word/token the LLM generates is sent as a JSON event: {"type":"token","content":"..."}. The browser appends it to the answer bubble in real time.

3. Done Event

When generation is complete, a final event {"type":"done",...} carries images, captions, verification status, and the complete answer. Images appear after the text finishes.

4. Error Event

If any error occurs mid-stream, an {"type":"error"} event is sent and the connection closes gracefully. The user sees the error inline in the chat.

Technical Implementation

  • Server: FastAPI StreamingResponse with media_type="text/event-stream" and Cache-Control: no-cache headers
  • LLM streaming: LangChain's llm.astream(messages) async generator — yields one AIMessageChunk per token
  • Retrieval in async context: Synchronous retrieval functions run in a thread pool via asyncio.to_thread() so they don't block the async event loop
  • Browser: Native ReadableStream API reads the SSE connection; no external library needed
Section 08

Chat Memory & Session Context

The system maintains conversation history within a session, enabling natural multi-turn conversations where users can ask follow-up questions without repeating context.

How Memory Works

  • The browser maintains a chatHistory array of all messages in the current session
  • Every API request includes the last 4 turns (8 messages: 4 user + 4 assistant) of history
  • History is capped at 1,800 characters to fit within LLM context limits without wasting tokens on old conversation
  • The query rewriting step (§5.1) uses this history to resolve pronouns and follow-up references before retrieval

Example Multi-Turn Conversation

User: When was Nalanda Mahavihara established?

AI: Nalanda Mahavihara was established in the 5th century CE during the Gupta period...


User: Who were its most famous scholars?

[System rewrites to: "Famous scholars of Nalanda Mahavihara 5th century CE Gupta period"]

AI: Among the most renowned scholars at Nalanda were Nagarjuna, Aryabhata, Dharmakirti...


User: Tell me more about the last one

[System rewrites to: "Dharmakirti Buddhist philosopher Nalanda contributions"]

AI: Dharmakirti was a 7th-century Buddhist logician and epistemologist...

Session scope: Chat history is stored in the browser's memory for the current session only. It is not persisted to a database — refreshing the page starts a fresh conversation. This is intentional for privacy and simplicity.

10.3 Web URL Sources — ChatGPT / Claude Browsing Approach

Admins can register external web pages as additional knowledge sources from the Web URL Sources section of the Knowledge tab. Unlike PDFs (which are chunked, embedded, and searched semantically), URL content uses a fundamentally different strategy — the same approach used by ChatGPT browsing and Claude's document reading: fetch at query time, inject directly into the LLM prompt.

Key insight: Web pages are typically small enough to fit in the LLM's context window in full. Rather than compressing them through vector search (which would lose detail), the system gives the LLM the entire page to read — just like you would paste a webpage into ChatGPT and ask a question about it.

How It Works

Step 1 — Admin Registers a URL

Admin enters any https:// URL in the admin panel. The server validates the URL is reachable (a quick HEAD request), then saves it to the url_sources database table. This is instant — no content is fetched or processed at add time.

Step 2 — Fetched Fresh at Query Time

When any user sends a chat message, all admin-registered URLs are fetched in parallel (via ThreadPoolExecutor). HTML is stripped using BeautifulSoup, leaving clean readable text. Fetched content is cached for 1 hour — subsequent queries within that window use the cached copy instantly.

Step 3 — Injected Directly into LLM Prompt

URL content does not go through the vectorstore or semantic search. It is appended directly to the LLM's prompt as a clearly labelled "Additional web sources" section — alongside the PDF context retrieved from ChromaDB. The LLM reads and reasons over both sources simultaneously.

Step 4 — LLM Synthesises Both

The LLM generates an answer using both the most relevant PDF chunks from vector search AND the full text of each registered web page. If a web page contains more relevant information than the PDF knowledge base for a particular question, the LLM will naturally use it.

Section 11

Authentication & Security

Authentication Flow

  • Login sends credentials to POST /api/login over HTTP
  • Server verifies password against bcrypt hash stored in SQLite
  • On success, a cryptographically secure random token (32 bytes, URL-safe base64) is generated with Python's secrets.token_urlsafe()
  • Token is stored server-side in an in-memory session dictionary keyed to user info
  • All subsequent API calls include the token in the Authorization: Bearer <token> header
  • Logout calls POST /api/logout which removes the token from the session store immediately

Authorization

Public Endpoints

/api/login, /api/signup, /api/health — no authentication required

User Endpoints

/api/chat, /api/chat/stream, /api/logout — any authenticated session (user or admin)

Admin Endpoints

All /api/admin/* endpoints — only sessions where role == "admin". Returns 403 Forbidden otherwise.

Password Security

  • All passwords are hashed with PBKDF2-HMAC-SHA256 (260,000 iterations) and a unique per-user random salt before being stored in the database
  • Plain-text passwords are never written to disk or logs
  • Password validation enforces minimum 6 characters during sign-up
  • Username must be 3–32 characters; email must contain @

File Upload Security

  • Only .pdf files are accepted by the upload endpoint — other file types return 400 Bad Request
  • Filename is sanitized using Path(filename).name to strip any directory traversal components
  • Only admin-role sessions can upload files
Production recommendation: For production deployment, add HTTPS (TLS) via a reverse proxy such as Nginx or Caddy. The session tokens are secure random strings but should be transmitted over encrypted connections. Also consider setting SESSION_TIMEOUT to expire inactive sessions automatically.

Section 14

LLM Provider Switching

The system supports multiple LLM providers and can be switched without any code changes — only .env file updates are needed. This prevents vendor lock-in and allows cost optimization.

Supported Providers

🟢

OpenAI (Default)

GPT-4o used for generation, rewriting, verification, and HyDE. text-embedding-3-small for embeddings. Best overall quality.

🔵

Anthropic Claude

Claude Opus/Sonnet used as drop-in replacement for all LLM calls. Set LLM_PROVIDER=anthropic and provide ANTHROPIC_API_KEY.

⚡

Groq (Partial)

Used specifically for text summarization during indexing when GROQ_API_KEY is set. Very fast inference for batch summarization jobs.

Provider Priority Logic

  • LLM for generation/RAG: checks LLM_PROVIDER → uses Anthropic if set to "anthropic" with valid key, otherwise OpenAI
  • LLM for summarization: checks Anthropic first → then Groq → then OpenAI (falls through to always find a working provider)
  • Embeddings: checks EMBEDDING_BACKEND → HuggingFace (local, free) or OpenAI API
  • Re-ranking: always local (sentence-transformers cross-encoder, no API cost)

Embedding Backend Options

OpenAI Embeddings (Default)

Uses text-embedding-3-small API. Highest quality embeddings. Requires API calls per document during indexing and per query during retrieval. Small cost per query.

HuggingFace Embeddings

Uses sentence-transformers/all-MiniLM-L6-v2 running locally on CPU. Zero API cost. Slightly lower quality than OpenAI. Good for development or budget deployments. Set EMBEDDING_BACKEND=huggingface.

Important: Switching embedding backends requires a full re-index of all documents. The embedding vectors in ChromaDB must be generated by the same model used at query time, otherwise similarity scores will be meaningless.
Section 15

Glossary

RAG (Retrieval-Augmented Generation)
An AI technique where the model retrieves relevant documents before generating an answer, grounding its response in real evidence rather than relying on training data alone.
Agentic RAG
An advanced form of RAG where the pipeline is orchestrated as a state machine (agent) that can make decisions — such as re-trying with a refined query or triggering web search — rather than following a fixed linear flow.
LangGraph
A framework built on top of LangChain for creating stateful, graph-based AI workflows. Each node in the graph is a function (retrieve, generate, verify), and edges define the flow including conditional branching.
Embedding / Vector Embedding
A numerical representation (list of hundreds of numbers) of text that captures its semantic meaning. Texts with similar meanings have embeddings that are mathematically close to each other, enabling similarity search.
ChromaDB
An open-source vector database that stores embeddings and enables fast nearest-neighbor similarity search. Uses SQLite as its storage backend.
Multi-Vector Retriever
A retrieval pattern where LLM-generated summaries (small, queryable) are stored in the vector store, but the full original documents (large, context-rich) are stored separately. Queries match against summaries; full documents are returned as context.
Cross-Encoder (Re-Ranking)
A neural network that reads a query and a document together to score their relevance. More accurate than embedding similarity but slower — used after initial retrieval to reorder candidates.
HyDE (Hypothetical Document Embeddings)
A retrieval technique where the LLM generates a hypothetical answer to the query, and that hypothetical answer's embedding is used for search — because it is closer in embedding space to real document passages than the query alone.
CRAG (Corrective RAG)
A technique that checks the quality of retrieved documents and falls back to an alternative source (web search) if the local knowledge base doesn't have sufficient information.
Self-RAG
A technique where the RAG system uses the LLM to verify its own generated answer against the retrieved context, flagging answers that are not grounded in the evidence.
SSE (Server-Sent Events)
A web standard for one-way, server-to-browser data streaming over a standard HTTP connection. Used here to stream AI-generated tokens to the browser in real time.
Docstore
The storage layer holding full parent documents (original text, table HTML, base64 images) indexed by UUID. Separate from the vector store which holds only summaries.
OCR (Optical Character Recognition)
Technology that converts images of text (scanned pages) into machine-readable text. Used by Tesseract to make scanned PDFs searchable.
bcrypt
A cryptographic password hashing algorithm designed to be slow and resistant to brute-force attacks. All user passwords are stored as bcrypt hashes.
Bearer Token
An authentication credential sent in the HTTP Authorization header as Bearer <token>. The server validates the token against its in-memory session store on every request.
Base64
A text encoding for binary data (like images). PDF-extracted images are stored as base64 strings and returned to the browser which decodes and displays them as <img> elements.
Chunk
A portion of text extracted from a PDF page. A single PDF is split into many chunks during ingestion — each chunk is a logical section of content (paragraph, sub-section, or table). Chunks are the fundamental unit of the knowledge base: each one is summarized, embedded, and stored separately. A 100-page PDF typically produces 50–200 chunks depending on layout and density.
Chunk Boundary / by_title Strategy
The rule that decides where one chunk ends and the next begins. This system uses the "by_title" strategy — chunks are split at section headings (e.g., "Chapter 3", "Excavation Results") rather than at arbitrary character counts. This keeps related content together and prevents answers from being split mid-thought. Three parameters control chunk size: CHUNK_MAX_CHARS (hard ceiling), CHUNK_NEW_AFTER (soft split point), and CHUNK_COMBINE_UNDER (merge short adjacent chunks).
CHUNK_MAX_CHARS (default: 10,000)
The absolute maximum number of characters a single chunk can contain. Any section longer than this is forcibly split. Set this lower (e.g., 4,000) if you want more granular retrieval; higher if your documents have very long sections that should stay together.
CHUNK_COMBINE_UNDER (default: 2,000)
If a chunk is shorter than this value, it is merged with the adjacent chunk rather than stored alone. Prevents tiny one-sentence chunks that would produce poor embeddings and unhelpful context for the LLM.
CHUNK_NEW_AFTER (default: 6,000)
The soft target size for starting a new chunk. Once a chunk reaches this length, the next section heading causes a split. Works together with CHUNK_MAX_CHARS to balance chunk size.
Summary Vector
The numerical embedding of the LLM-generated summary of a chunk — not the embedding of the raw chunk text. Stored in ChromaDB. When a user asks a question, their query is also embedded and compared against these summary vectors to find the most semantically similar chunks. The count shown as "Summary Vectors" in the admin panel is the total number of these entries in the database.
Docstore Entry
The full original content of a chunk (complete text, table HTML, or base64 image) stored in docstore.pkl, linked to its summary vector by a UUID. When retrieval finds a matching summary vector, the system fetches the corresponding docstore entry and sends the full content to the LLM as context. The "Docstore Entries" count in the admin panel reflects how many parent documents are stored.
Vector Store Path
The server directory where ChromaDB persists its SQLite database (chroma.sqlite3) and the docstore pickle file (docstore.pkl). Set via VECTORSTORE_PATH in .env or overridden per-session in the admin panel's Advanced Paths field. Multiple paths = multiple independent knowledge bases.
Parent Document
The full original content stored in the docstore, as opposed to the summary stored in the vector database. The "parent" of a summary vector — when retrieval finds a summary, the parent is fetched and returned. Contains the complete text/table/image that the LLM actually reads to generate its answer.
Summary Vectors vs Docstore Entries
These two numbers in the admin stats should be close but not necessarily equal. Summary Vectors = searchable index size (how many things the AI can find). Docstore Entries = storage size (how many full documents are retrievable). Images may have docstore entries without separate summary vectors if they share a vector with their parent text chunk.
Single PDF Mode
Indexing mode where only one selected PDF is processed. The system first removes all existing entries for that file (by matching the source path), then re-indexes it fresh. All other documents in the knowledge base are untouched. Ideal for updating one changed document.
Entire Folder Mode
Indexing mode where every PDF in the configured data folder is processed sequentially. For each file, existing entries are removed before new ones are added — making this safe to re-run without creating duplicates. Suitable for initial full knowledge base population or bulk updates.
Advanced Paths
Optional admin panel fields to override the server's default storage paths for a specific indexing job. Vector store path changes where ChromaDB writes/reads. Data folder path changes which folder is scanned for PDFs. Leaving both blank uses the server's .env defaults. Useful for maintaining separate test and production knowledge bases on the same server.
Background Job
Indexing always runs as a background thread on the server — it does not block the web interface or other users from chatting while it runs. The admin panel polls the job status every 1.5 seconds and shows live log output. The Index button is disabled while a job is running to prevent duplicate submissions.
Web URL Sources
Admin-configured web page URLs stored in the url_sources database table. Unlike PDFs, they are not indexed into the vector store. Instead, their content is fetched at query time and injected directly into the LLM's prompt alongside PDF context — the same approach used by ChatGPT browsing and Claude document reading. Useful for live reference pages, official documentation, or any web content that should always be available to the AI regardless of the user's question.
URL Content Cache
An in-process server-side cache (Python dictionary) that stores fetched web page content for 1 hour after the first fetch. When a user asks a question, each registered URL is checked: if a fresh cached copy exists it is used instantly; otherwise the page is fetched from the internet. The cache is invalidated immediately when a URL is deleted from the admin panel. Prevents redundant HTTP requests on every user query.
In-Prompt Web Context (vs. Vector Search)
The distinction between how PDFs and URLs are given to the LLM. PDFs go through semantic retrieval — only the most relevant chunks are selected from thousands of possibilities and given to the LLM. URL content goes through direct prompt injection — the full page text is always included, regardless of the question. Both appear in the same LLM prompt, and the LLM synthesises an answer from both simultaneously. Use PDFs for large corpora that need selective retrieval; use URL sources for small reference pages that should always be available in full.
PBKDF2-HMAC-SHA256
The password hashing algorithm used by this system. PBKDF2 (Password-Based Key Derivation Function 2) applies the SHA-256 hash function 260,000 times with a unique random salt per user. This makes brute-force and rainbow-table attacks computationally infeasible. Each user's salt and hash are stored in the database; the original password is never stored.
